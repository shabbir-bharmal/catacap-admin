Release 19_05_2026
==================

This release bundles two independent change sets:
  A. Referrals admin module registration
  B. Cover Fees: hold-on-create for Pending Grants / Other Assets


A. Referrals admin module
-------------------------

Migrations
----------
1. 2026_05_19_add_referrals_module.sql
   - Schema: public.modules
   - Intent: Register the 'referrals' admin module row so the new Referrals
     admin page (under "Donations to Invest") can be wired into the role /
     module access permissions system alongside Users, Recommendations,
     Pending Grants, and Other Assets.
   - Idempotency: Guarded by `WHERE NOT EXISTS (SELECT 1 FROM public.modules
     WHERE name = 'referrals')`. Safe to re-run.
   - Rollback:
       DELETE FROM public.module_access_permissions
        WHERE module_id = (SELECT id FROM public.modules WHERE name = 'referrals');
       DELETE FROM public.modules WHERE name = 'referrals';

Notes
-----
- The `public.referrals` table itself and `public.users.ref_code` already exist
  in the database and are not changed by this release.
- The Referrals admin page is read-only; no schema/data writes are performed.


B. Cover Fees — hold-on-create for Pending Grants / Other Assets
================================================================

Migrations (apply in order)
---------------------------
1. 2026_05_19_001_cover_fees_hold_status.sql
   - Adds `status` (TEXT, default 'applied', CHECK in ('held','applied'))
     to campaign_cover_fees_activity.
   - Adds FK columns:
        * triggered_by_pending_grant_id              -> pending_grants(id)               ON DELETE SET NULL
        * triggered_by_asset_based_payment_request_id -> asset_based_payment_requests(id) ON DELETE SET NULL
   - Adds partial unique indexes guaranteeing at most one ACTIVE
     (fully_reversed_at IS NULL) activity row per (pool, pending_grant)
     and per (pool, asset_based_payment_request).
   - Adds `idx_ccfa_held` for fast SUM-by-pool of currently-held escrow.
   - Idempotent: all DDL guarded by IF NOT EXISTS / DO blocks.

2. 2026_05_19_002_cover_fees_holds_backfill.sql
   - Materializes 'held' rows for every Pending pending_grant / asset
     payment request linked to an active capped pool.
   - Materializes 'applied' rows (incrementing pool.amount_used) for
     every In Transit row that doesn't already have an active activity
     row. Where a recommendation already exists for the pending grant
     it is linked via triggered_by_recommendation_id.
   - First-come-first-served by request creation date, single-cover
     policy (one pool per request), respects coverage_active_from.
   - Idempotent: re-running the migration is a no-op (NOT EXISTS guards
     plus partial unique indexes from migration 001).

Runtime behavior added by this release
--------------------------------------
* Cover Fees activity rows now carry one of two statuses:
    - 'held'    : pool escrow reserved, amount_used NOT yet incremented.
    - 'applied' : pool escrow drawn, amount_used incremented, donor /
                  sponsor Account-History row written.

* server/src/utils/coverFees.ts
    - `holdCoverFeeForRequest({ requestKind, requestId, ... })`
        Places a 'held' activity row against the first eligible pool
        (matches campaign, active, coverage_active_from satisfied,
        budget remaining after existing amount_used + outstanding holds).
        Idempotent — re-invoking on an already-held request is a no-op.

    - `convertHeldCoverFeeToApplied({ requestKind, requestId,
        triggeringRecommendationId })`
        Flips status='held' -> 'applied', increments pool.amount_used,
        backfills the recommendation FK, and writes the same
        "Cover Fees - escrow applied" donor / sponsor Account-History
        row that applySingleCoverFee writes today. Returns the converted
        fee amount (0 if no held row existed for this request).

    - `releaseHeldCoverFeeForRequest({ requestKind, requestId })`
        Deletes any active 'held' activity rows for the request.
        Used on cancel / reject / soft-delete of a still-Pending request.
        Holds never touched the sponsor's wallet (escrow only), so
        releasing simply frees pool capacity — no Account-History row
        is written.

    - `backfillRecommendationLinkForRequest({ requestKind, requestId,
        triggeringRecommendationId })`
        Sets triggered_by_recommendation_id on an existing 'applied'
        row when the rec is created strictly after the row was applied
        (Other Assets In Transit -> Received transition).

    - `reconcilePendingHoldsForAllPools()`
        Runs the same logic as the backfill migration on demand to
        catch new requests created by the .NET backend (the Node side
        does not own the create path). Called from
        GET /api/admin/cover-fees so the admin always sees an up-to-date
        "On Hold" amount.

* Existing applyCoverFees / applySingleCoverFee continue to fire on
  Pending -> In Transit transitions when the row was created BEFORE the
  cover-fees pool existed (no hold to convert). When a hold IS present,
  convertHeldCoverFeeToApplied runs instead and the fire-and-forget
  applyCoverFees becomes a no-op (idempotent skip on the unique index).

* Pool "Remaining" everywhere now subtracts outstanding 'held' escrow
  alongside amount_used so the UI never overstates pool capacity.

* AdminCoverFees.tsx — Edit Cover Fees Pool page now shows an
  "On Hold" row alongside "Already covered" / "Remaining" so the sponsor
  can see escrowed-but-not-yet-drawn amounts at a glance.

* Pending fee coverage projection (server/src/utils/pendingCoverFees.ts)
  no longer hides pending grants / DAF requests that already have a
  'held' activity row — those are exactly what the "Pending fee
  coverage" table is meant to surface. Triggers with a held row are
  rendered using the actual stored fee_amount and do not double-count
  against pool remaining capacity (heldSum already subtracted once).

C. Referrals: manual "Add Referral" with retroactive back-fill
--------------------------------------------------------------

Migration
---------
* 2026_05_19_003_referrals_unique_signup.sql
  Adds partial unique index
    idx_referrals_signup_unique_referred_user
    ON public.referrals(referred_user_id) WHERE action_type='signup'
  so a referred user can have at most ONE 'signup' attribution row.
  This is the authoritative DB-level guard against two admins
  concurrently linking the same person to two different referrers.
  Pre-flight DO-block aborts the migration if existing data already
  contains duplicate signup rows. Idempotent (IF NOT EXISTS).

Intent
------
* Add an "Add Referral" button on /referrals that opens a dialog with
  two email-pickers (referrer + referred). Submitting calls the new
  endpoint POST /api/admin/referrals/link.

* The endpoint inserts a signup attribution row AND back-fills
  retroactive activity rows for everything the referred user has
  already done:
    - group_join: public.requests where request_owner_id = referred,
      status='accepted', non-deleted, group_to_follow_id NOT NULL.
    - investment: public.recommendations where user_id = referred,
      non-deleted, campaign_id NOT NULL (amount carried over).
    - raise_money_signup: public.campaigns where user_id = referred,
      non-deleted.

* Every insert uses NOT EXISTS (signup uses ON CONFLICT DO NOTHING on
  the new unique index) so re-submitting the same pair only inserts
  rows that aren't there yet — totalNew=0 means "already fully linked".

* Conflict handling: if the referred user already has a signup row
  pointing at a different referrer, the endpoint returns 409 with
  "This user is already attributed to a different referrer." The same
  message is returned both pre-flight and on the unique-violation race
  path inside the catch block.

Rollback
--------
  DROP INDEX IF EXISTS public.idx_referrals_signup_unique_referred_user;

Verification
------------
1. psql -f releases/19_05_2026/migrations/2026_05_19_003_referrals_unique_signup.sql
2. Verify index exists:
     SELECT indexname FROM pg_indexes
      WHERE schemaname='public'
        AND indexname='idx_referrals_signup_unique_referred_user';
3. /referrals → "Add Referral" → pick two users → Save Referral.
   Toast shows counts; the new referrer appears (or its counts grow)
   in the table after auto-invalidate.
4. Submit the same pair again → toast "Referral already linked" with
   0 new events.
5. Pick a referred user that already belongs to another referrer →
   error toast with the 409 message; no rows inserted.


D. Referrals: ensure every user has a ref_code
----------------------------------------------

Migration
---------
* 2026_05_19_004_users_ref_code_backfill.sql
  - Creates helper function public.generate_user_ref_code() that
    returns a random 8-char alphanumeric code (a-z, A-Z, 0-9) which
    is not already present in public.users.ref_code.
  - Back-fills ref_code for every user where it is NULL or '' using
    the helper (loops in PL/pgSQL, one UPDATE per missing user).
  - Adds partial unique index idx_users_ref_code_unique
      ON public.users(ref_code) WHERE ref_code IS NOT NULL AND <> ''
    so two users can never share a code.
  - Idempotent: function is CREATE OR REPLACE; backfill targets only
    NULL/empty rows; index uses IF NOT EXISTS.

Intent
------
* Previously ref_code was only populated for a handful of legacy users
  (e.g. Ken Kurtzig), so the /referrals admin table showed "—" for
  almost everyone and the public site had no referral link to give
  out. After this release every account in public.users has a code,
  and every user created through the Node backend is born with one.

Runtime changes
---------------
* server/src/utils/refCode.ts (new)
    generateUniqueRefCode(client?) — produces an 8-char alphanumeric
    code (crypto.randomBytes -> CHARS lookup), retried up to 12 times
    until SELECT confirms the code is not already in users.ref_code.
    Pass an open PoolClient when generating inside a transaction.

* server/src/utils/anonymousUser.ts
    findOrCreateAnonymousUser now generates a ref_code inside the
    insert transaction. On a 23505 hit against the new unique index
    it retries up to 5 times with a fresh code before failing.

* server/src/routes/adminUsers.ts (POST /api/admin/users — create user)
    Same pattern: generate ref_code, INSERT with it, retry on
    23505 / idx_users_ref_code_unique up to 5 times. On exhaustion
    returns 500 with "Could not allocate a unique referral code."

Caveat
------
* The .NET backend (Back-End/) also creates users via the public
  registration / OAuth paths and is OUT OF SCOPE for edits per the
  project's user-preference rule. Users created there will land with
  ref_code = NULL until either the .NET side is patched OR the
  back-fill migration is re-run (the back-fill is idempotent and will
  populate any newly-NULL rows on a second invocation).

Rollback
--------
  DROP INDEX IF EXISTS public.idx_users_ref_code_unique;
  DROP FUNCTION IF EXISTS public.generate_user_ref_code();
  -- (existing ref_code values left intact; harmless if kept)

Verification
------------

2. SELECT COUNT(*) FILTER (WHERE ref_code IS NULL OR ref_code = '')
     FROM public.users;
   Expect 0.
3. /referrals → REF CODE column shows a code for every row including
   Lily Kanter and Tim Freundlich.
4. POST /api/admin/users (admin "Create user" flow) → newly created
   user row has a populated ref_code.
